一个使用 `Python + Pygame + OpenCV` 重构街机游戏《e舞成名》流程与演出表现的项目。

这不是单纯的界面临摹，也不是只做几个静态页面的展示工程。  
这个项目的重点，是把街机游戏里一整套连续流程拆开后重新实现，包括场景切换、界面组织、视频演出、谱面播放和结算反馈，并尽量保留原作那种“厅机流程感”。

目前它还处在持续开发阶段，不是完整成品，但主流程骨架已经搭起来了，而且很多模块已经能单独调试和反复迭代。

---

## 项目目标

这个项目主要在做三件事：

- 用 Python 重建《e舞成名》的主流程与场景结构
- 尽可能还原原作的交互节奏、界面气质和演出方式
- 把原本偏一次性的实现，整理成能继续维护和扩展的工程结构

它更接近“街机流程重构”，而不是一个只模仿视觉风格的小游戏壳子。

---

## 当前已经实现的内容

### 主流程场景

目前已经拆分并实现了这些独立场景：

- `投币`
- `玩家选择`
- `登陆磁卡`
- `个人资料`
- `大模式`
- `子模式`
- `选歌`
- `加载页`
- `谱面播放器`
- `结算`

这些场景已经能按主流程衔接，形成一套完整的基础链路。

### 演出与渲染

- 使用 `OpenCV` 处理开场动画和菜单背景视频
- 使用 `Pygame` 统一接管场景渲染
- 支持黑屏过渡、点击特效、顶部栏、底部状态栏、全局 BGM 管理等街机常见表现

### 谱面播放相关

- 独立谱面播放器场景
- 准备就绪动画
- HUD 渲染
- 击中特效
- 判定、计分、血条等基础玩法模块

### 数据与配置

- 多数页面布局和参数已经拆到 `json/` 下管理
- 包括全局设置、加载页、选歌、个人资料、结算、谱面渲染器等配置
- 便于调整界面布局，而不是把坐标全部写死在代码里

### 调试工具

项目里已经整理了不少专项调试入口，方便单独调某个页面或模块，而不用每次从完整主流程一路点进去。

## 这个项目的特点

这个仓库比较有意思的地方，不在于“像不像原版截图”，而在于它已经开始处理街机项目里那些真正麻烦的部分：

- 多场景切换和状态流转
- 视频背景与音频管理
- 以设计稿坐标为基础的布局映射
- 资源驱动的 UI 组织方式
- 谱面渲染和游戏内 HUD
- 个人资料、信用值、模式选择这类非核心玩法系统
- 为了持续开发而准备的调试器和测试入口

所以它更像一台正在被 Python 重构中的街机系统，而不是一个只做表面效果的 demo。

## 技术栈

- `Python 3`
- `Pygame`
- `OpenCV`
- `JSON` 配置驱动

## 目录结构

- `main.py`  
  主入口，负责窗口初始化、场景切换、全局投币逻辑、热重载、背景视频和调试器接线

- `scenes/`  
  各个场景实现，从投币到结算的完整流程都在这里

- `ui/`  
  通用 UI 组件与表现模块，包括顶部栏、点击特效、布局调试器、谱面渲染器、准备就绪动画等

- `core/`  
  核心基础模块，包括字体、音频、视频、路径、工具函数，以及部分玩法子模块

- `json/`  
  各类布局和配置文件，包括全局设置、加载页、选歌、结算、个人资料、谱面渲染器等

- `UI-img/`  
  界面资源与图集，例如按钮、特效、经验条、血条、准备就绪、结算资源等

- `backmovies/`  
  开场和菜单循环背景视频

- `冷资源/`  
  字体、背景图、背景音乐、按钮音效等资源目录

- `songs/`  
  歌曲数据目录，包括索引与不同模式下的谱面数据

- `调试器归档-要用拉到根目录/`  
  单模块专项调试脚本，例如加载页、结算、谱面布局、圆环频谱、升级动画等

- `测试入口归档-要用拉到根目录/`  
  独立测试入口，可直接跳转到指定场景进行验证

- `设计图/`  
  设计稿和部分参考资源

---

#### 运行方式

先安装依赖：

```bash
pip install pygame opencv-python
```

然后运行主入口：

```bash
python main.py
```

如果本地资源完整，程序会从开场动画进入主流程，默认从投币场景开始。

## 技术说明

### 场景系统

项目基于 `场景基类.py` 管理多场景逻辑，负责场景切换、状态传递和主循环调度。

### 谱面系统

谱面相关逻辑已经拆分为独立模块，包括解析、渲染、判定、计分等部分，便于单独维护和继续细化。

### UI 系统

UI 部分不是纯代码硬写，已经开始向“资源 + 配置驱动”的方向整理，包括布局映射、动画表现和交互效果。

### 配置驱动

项目大量使用 JSON 管理页面布局、参数和部分表现逻辑，主要是为了提高迭代效率。
这套方式的优点是改布局比较直接；缺点也很明显：配置一多以后，维护成本会上升，所以后续还需要继续规范。
